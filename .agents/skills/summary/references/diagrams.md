# Diagram Guide

Diagrams are supporting material for one-paper summaries. Use them when the reader would need more effort to reconstruct structure from prose.

## Selection

| Situation | Template |
|---|---|
| System components, ownership, boundaries | `architecture.html` |
| Branching decision or workflow | `flowchart.html` |
| Priority, positioning, value vs effort | `quadrant.html` |
| Category comparison | `bar-chart.html` |
| Trend across time | `line-chart.html` |
| Share of whole, up to 6 parts | `donut-chart.html` |
| Lifecycle or status transitions | `state-machine.html` |
| Roadmap or milestones | `timeline.html` |
| Cross-team or cross-role process | `swimlane.html` |
| Hierarchy | `tree.html` |
| Layered architecture or stack | `layer-stack.html` |
| Overlap between sets | `venn.html` |
| OHLC price movement | `candlestick.html` |
| Positive and negative contribution bridge | `waterfall.html` |

## Rules

- One paper usually uses no more than one diagram.
- A diagram should carry one insight.
- Use short labels.
- Keep color within the paper token set.
- Put detailed caveats in surrounding prose.
- Captions state the point the reader should remember.

## Redrawn Sources

When redrawing a third-party paper figure, patent illustration, official architecture diagram, or public chart, add `示意重绘` / `Schematic redrawn` in the caption.
